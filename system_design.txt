page-level OCR with document-level aggregation.

Reason:

PDFs naturally decompose into pages
OCR latency often scales with page complexity
page-level retries are much cheaper than redoing an entire 200-page document

The tradeoff is worth it because it gives much better scaling and failure isolation.

The wrong design is to log everything by job_id in metrics. That creates high-cardinality problems.

The better design is:

metrics for aggregates
traces for per-job detail
logs for event context
DB for business state

Prometheus recommends histograms for latency distributions, and current Prometheus guidance prefers native histograms for new instrumentation when supported.

So I would use:

Metrics

upload rate
queue depth
page OCR latency
classification latency
failure rate by stage
callback success rate
model version request counts

Tracing

one trace per job
spans for upload, preprocessing, OCR, classification, persistence, callback

Logs

structured logs with job_id, document_id, tenant_id, model_version

This division is the right tradeoff because:

metrics are cheap and alertable
traces help debug slow or failed jobs
logs provide operational detail without exploding metric cardinality
